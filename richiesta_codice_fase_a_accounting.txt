Richiesta codice per Fase A - Accounting e ownership memoria raster/WebGL

Abbiamo deciso di partire dalla Fase A: accounting centrale e ownership esplicita per texture/FBO.

Ti chiedo una proposta concreta di codice/patch, non solo indicazioni architetturali.

Obiettivo:
implementare un primo ResourceManager/MemoryManager leggero che tracci tutte le risorse raster/WebGL principali e permetta di generare un report memoria affidabile.

Puoi fornire il codice necessario per introdurre:

1. ResourceManager centrale
   - registrazione texture;
   - registrazione framebuffer/renderbuffer se utile;
   - unregister/delete esplicito;
   - update dei metadata;
   - stima byte per risorsa;
   - conteggi create/delete;
   - rilevazione risorse orphan/suspect quando possibile.

2. Metadata minimi per ogni texture
   - id risorsa;
   - ownerId;
   - ownerType: live, cache, scratch, historyGpu, historyCpuRaw, historyCompressed, orphan/suspect;
   - kind: layer, previewMip, effectScratch, strokeScratch, historySnapshot, backdrop, transformPreview, paintTarget, other;
   - width;
   - height;
   - format;
   - bytes;
   - bbox/origin se disponibile;
   - isFullCanvas;
   - bboxCoverage;
   - purgeable;
   - reason;
   - createdAt;
   - lastUsed;
   - stack/debug tag opzionale.

3. API pubblica iniziale
   - registerTexture(texture, metadata);
   - updateTexture(texture, metadataPatch);
   - deleteTexture(textureOrId);
   - registerFramebuffer(framebuffer, metadata);
   - deleteFramebuffer(framebufferOrId);
   - markUsed(textureOrId);
   - reportRasterMemory();
   - getTopResourcesByBytes(limit = 20).

4. Report richiesto da window.CBO.reportRasterMemory()
   Deve distinguere almeno:
   - totalEstimatedGpuBytes;
   - liveLayerBytes;
   - previewCacheBytes;
   - scratchBytes;
   - effectScratchBytes;
   - strokeScratchBytes;
   - blendBackdropBytes;
   - transformPreviewBytes;
   - historyGpuBytes;
   - historyCpuRawBytes;
   - historyCompressedBytes;
   - orphanSuspectBytes;
   - textureCount;
   - framebufferCount;
   - renderbufferCount;
   - createdTextureCount;
   - deletedTextureCount;
   - fullCanvasResourceCount;
   - fullCanvasMaterializationCount;
   - topResourcesByBytes.

5. Logging obbligatorio full-canvas
   Ogni volta che una risorsa viene materializzata full-canvas, vogliamo tracciare:
   - tool;
   - layerId;
   - oldBBox;
   - newSize;
   - reason;
   - bytesAdded;
   - stack/debug tag.

6. Integrazione incrementale
   Non serve riscrivere tutto il renderer.
   Vorrei una patch pragmatica che possa essere inserita sopra le funzioni esistenti di creazione/cancellazione texture.
   Se non puoi vedere il repository, dammi comunque codice adattabile e indica chiaramente dove agganciarlo:
   - create texture layer;
   - create preview mip;
   - create scratch/effect target;
   - create history snapshot;
   - delete texture;
   - delete layer;
   - clear preview cache;
   - clear scratch.

7. Criteri di accettazione
   Dopo la patch dovremmo poter aprire un documento 4000x4000 e chiamare:

   window.CBO.reportRasterMemory()

   ottenendo una struttura leggibile con:
   - byte stimati per categoria;
   - top 20 texture per peso;
   - indicazione immediata delle texture full-canvas;
   - conteggio delle risorse purgeable;
   - risorse sospette/orfane;
   - differenza chiara tra live, cache, scratch e history.

Domanda principale:
puoi fornirmi direttamente il codice/patch per questa Fase A, con nomi di file suggeriti, API esposte e punti di integrazione nel renderer esistente?

Se servono assunzioni, falle esplicite. Preferisco una prima versione semplice ma funzionante, estendibile poi con trimRasterMemory e policy history.
